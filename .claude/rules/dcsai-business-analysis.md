## F&F DCS 용어

- 사용자 권한에 따라 조회 제한이 있을 수 있다

## 분석 실행계획 프로세스

### 실행계획 원칙
- **실행계획은 내부적으로 수립하고, 사용자 승인 없이 즉시 실행한다**
- 탐색 → 계획 수립 → 실행 → 검증 → 분석 → 결과 제시의 전체 흐름을 자율적으로 진행한다
- 사용자에게는 `데이터를 성공적으로 조회했습니다. 분석을 시작합니다` 메시지 이후 최종 분석 결과와 추천 액션을 제시한다
- **자율 진행 예외 (사용자 질의 필요)**:
  - 상위/하위 스킬 간 명백한 상충 → 사용자에게 어느 쪽을 따를지 질의
  - 대응 불가 → "프로세스팀에 요청하시겠습니까?" 질의

### Phase 1: 탐색 및 계획 수립 (내부 처리, 사용자 질의 없이 자율 진행)
0. 최상위 시작 노드의 스킬 확인 후 이후 노드 탐색 여부 판단 — **미구현 항목에 해당하면 즉시 "미구현 기능입니다" 답변 후 종료** (→ `dcsai-tools.md` "의도 탐색 전략 (점진적 탐색)" 참조)
1. 미구현이 아닌 경우, 의도 탐색 전략(점진적 탐색)에 따라 **최하위 노드까지** 인텐트/스킬/도구 탐색 (→ `dcsai-tools.md` "의도 탐색 전략" 및 "상위 의도 스킬 기본값 규칙" 참조)
1.1 노드 탐색시 ***반드시 INTENT가 가진 SKILL을 확인한다***
1.5. **기준정보 사전 검증** — 분석 의도 탐색 중 형제·보조 노드(기준정보 조회)가 존재하면, 분석 정확도를 위해 적극 활용한다:
   - **활용 판단 기준**: 사용자 질의에 특정 필터 조건(상품, 채널, 매장, 기간 등)이 포함되어 있거나, 도구 스키마의 파라미터 구성에 분류체계·유효값 확인이 필요한 경우
   - **보조 노드 유형별 활용**:
     | 조건 | 보조 노드 | 활용 |
     |------|----------|------|
     | 상품/카테고리/아이템 필터 포함 시 | 상품 정보 조회 → 상품 카테고리 분류체계 조회 | 분류체계 확인, 카테고리 범위 결정 (예: "아우터"와 "다운"이 별도 ITEM_GROUP인지) |
     | 채널/매장 필터 포함 시 | 판매채널 정보 조회 → 판매채널 분류체계 조회 | 채널 분류체계 확인, 정확한 채널 코드 확보 |
     | 특정 매장 언급 시 | 판매채널 정보 조회 → 매장 명칭/속성 기반 조회 | 매장코드(SHOP_ID) 확보 |
     | 도구 스키마의 필터 필드에 유효값 확인 필요 시 | 시스템 코드 조회 → 시스템 필드명의 유효값 조회 | 필터 파라미터 값 검증 |
     | 시즌/기간 분석 시 | 날짜 조회 | 정확한 시즌 기간, 동요일 기준 날짜, 시즌마감일 계산 |
   - **결과 활용**: 검증된 분류체계·코드값·날짜를 이후 Step 2의 파라미터 구성에 반영
2. 도구실행계획 수립 — 아래 항목을 내부적으로 정리한 뒤 즉시 Phase 2로 진행:
   - **누적 스킬 요약**: 계층별로 수집한 스킬 정리 (상위 스킬이 기본값으로 적용)
   - **필요 도구 목록**: 호출할 도구, 호출 순서, 병렬/순차 구분
   - **파라미터 구성**: 브랜드코드, 기간, selectors, metrics, filters 등 — **Step 1.5에서 검증된 값 사용**
   - **상위 스킬 지침 반영**: 전년비 비교, 채널 우선 등 상위 스킬에서 요구하는 사항 포함 여부 확인
3. 정합성 비교검증 계획수립 — 교차검증 방법(부분합 검증, 필터 최소화 검증 등) 결정

### Phase 2: 데이터 조회 및 검증 (내부 처리, 사용자 질의 없이 자율 진행)
4. 도구 호출 전 데이터 사이즈 점검 (→ `dcsai-tools.md` "데이터 사이즈별 처리 규칙" 참조)
5. 데이터 사이즈 별 호출방법 선택 (→ `dcsai-tools.md` "데이터 사이즈별 처리 규칙" 참조)
6. 데이터 조회 (계획에 따라 당해/전년 등 병렬 호출)
7. 교차검증 실행 및 결과 확인

### Phase 3: 분석 및 결과 제시 (사용자에게 출력)
8. `데이터를 성공적으로 조회했습니다. 분석을 시작합니다` 메시지 출력
9. 분석 실행
9-1. 시각화 판단: 사용자가 시각화를 요청했거나 분석 결과가 시각화에 적합한 경우
     → `dcsai-visualization.md` 매핑 테이블 참조하여 차트 유형 자동 선택
     → 차트 유형 명시 요청이 있으면 해당 유형 우선 적용
10. 결과 리뷰 및 추가 액션 제안


## 데이터 검증 원칙
- **교차검증 필수**: 분석 후 총합·부분합 검증
- 데이터 사이즈별 처리 방식(1500토큰 기준)은 → `dcsai-tools.md` "데이터 사이즈별 처리 규칙" 참조
- **1500토큰 미만** (context 로드): ***동일한 API***에 ***SELECTOR 최소화한 요청***을 병렬로 보내 결과값 교차비교
- **1500토큰 이상** (파일 저장): 코드 기반 검증 수행 (→ `data-processing` 스킬 교차검증 예시 참조)

### 검증 방법
1. **부분합 검증**: drill-down 결과의 합산 = 상위 총액 일치 확인
2. **필터 최소화 검증**: 필터 없이 전체 조회한 결과와 개별 필터 합산 비교
3. **결과 명시**: 통과 시 "교차검증 완료: 총합·부분합 일치 확인" 출력, 실패 시 불일치 내역 명시 후 재조회

## 분석 이후 추천액션
- 1순위 : 관련 노드 기반으로 추천
- 2순위 : 분석 결과 deep dive 문제 원인 유추
- 답변방식 : ordered_list 형식

## 대응 불가 시 처리
- 사용자 질의 대응 불가 시 → "프로세스팀에 요청하시겠습니까?" 질의
- 승인 시 `dcs-ai-common:Issue` 스킬로 이슈 리포트 전송
